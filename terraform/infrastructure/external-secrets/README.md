# external-secrets infrastructure

Bootstraps in-cluster secret delivery: the
[External Secrets Operator (ESO)][eso] syncs secrets from the 1Password vault
**`Frankenstructure`** into native Kubernetes `Secret`s, and
[Stakater Reloader][reloader] restarts workloads when those Secrets change.

This is the **foundation** stack for every other secret consumer. See the
[central architecture doc](../../../docs/architecture-config-secrets-cicd.md) for
how this fits the full config/secrets/CI-CD picture.

## What this deploys

| Resource                                | File                                | Purpose                                                                                          |
| --------------------------------------- | ----------------------------------- | ------------------------------------------------------------------------------------------------ |
| ESO operator + CRDs                     | `helm.tf`                           | `external-secrets` chart `2.7.0`, `installCRDs = true`.                                          |
| Stakater Reloader                       | `helm.tf`                           | `reloader` chart `2.2.12`; rolling-restarts annotated workloads.                                 |
| `ClusterSecretStore` **`onepassword`**  | `cluster_secret_store.tf` / `.yaml` | `onepassword` (Connect) provider ŌåÆ vault `Frankenstructure`.                                     |
| Secret-zero `onepassword-connect-token` | `secrets.tf`                        | The Connect API token ESO authenticates with (plus bootstrap `onepassword-connect-credentials`). |
| `external-secrets` namespace            | `namespaces.tf`                     | Holds all of the above.                                                                          |

**Secret-zero:** ESO cannot source its own auth token through itself
(chicken-and-egg), so the `onepassword-connect-token` Secret (plus the bootstrap
`onepassword-connect-credentials` the Connect server itself needs) is seeded
directly from Terraform variables, wired in CI to the corresponding GitHub Actions
secrets. Every other in-cluster secret derives from these.

## Runbook: add a new ExternalSecret

1. **Confirm it's a pure 1Password lookup.** If the value is generated by a
   provider at apply time (Vultr DB password, Honeycomb key, ŌĆ”) it stays
   Terraform-managed ŌĆö do **not** use ESO. See the
   [mixed-source rule](../../../docs/architecture-config-secrets-cicd.md#the-mixed-source-rule).

2. **Author the `ExternalSecret`** in the consuming stack (next to the workload),
   using the existing
   [`grafana-cloud`](../observability/external_secret_grafana_cloud.yaml) /
   [`miniflux-admin`](../../applications/miniflux/external_secret_miniflux.yaml) /
   [`automate-onepassword`](../../../fsharp/api/automate/terraform/external_secret_automate.yaml)
   CRs as templates:

   ```yaml
   apiVersion: external-secrets.io/v1
   kind: ExternalSecret
   metadata:
     name: my-secret
     namespace: my-namespace
   spec:
     refreshInterval: 1h
     secretStoreRef:
       name: onepassword
       kind: ClusterSecretStore
     target:
       name: my-secret # the k8s Secret ESO creates
       creationPolicy: Owner
     data:
       - secretKey: MY_KEY # key inside the produced Secret
         remoteRef:
           key: "Item Title" # the 1Password item title; vault from the store
           property: field # the field label on that item
   ```

   - **`remoteRef.key` = the item TITLE, `remoteRef.property` = the field label.**
     The `onepassword` (Connect) provider reads `key` as the item title and takes
     the field from a separate `property`. The vault is supplied by the
     ClusterSecretStore, so omit it. (Historical: the superseded `onepasswordSDK`
     provider had no `property` and slash-encoded the field into `key` as
     `"<item>/[section/]<field>"`.)
   - **`creationPolicy: Owner`** ŌĆö ESO owns the produced Secret's lifecycle.

3. **Apply via the kbst `kustomization_resource` pattern** ŌĆö never
   `kubernetes_manifest` (see [Gotchas](#gotchas)). Mirror the existing
   `cluster_secret_store.tf` / `external_secret_*.tf` files:

   ```hcl
   data "kustomization_overlay" "my_secret" {
     resources = ["${path.module}/external_secret_my_secret.yaml"]
   }

   resource "kustomization_resource" "my_secret" {
     for_each   = data.kustomization_overlay.my_secret.ids
     manifest   = data.kustomization_overlay.my_secret.manifests[each.value]
     depends_on = [kubernetes_namespace.my_namespace]
   }
   ```

4. **Match names to the consumer.** The produced Secret's `target.name` and each
   `secretKey` must match exactly what the workload references (via `envFrom`
   `secret_ref`, a volume, etc.). Workloads use `optional = false` so a missing
   Secret fails loudly.

5. **Remove the now-duplicated CI wiring.** Delete the corresponding
   `TF_VAR_*` env and `op://ŌĆ”` ref from that app's deploy job in
   `.github/workflows/cicd.yaml` ŌĆö the value now flows through ESO, not the job.

6. **Ensure deploy ordering.** App deploy jobs already gate on
   `deployed-foundation` (ŌŖć `deploy-external-secrets`), so the store exists before
   the ExternalSecret reconciles. For a **foundation-tier** stack (like
   observability) add an explicit `needs: deploy-external-secrets` to its job.

7. **Enable rotation (optional).** If the workload must pick up rotated values
   without a manual rollout, annotate it `reloader.stakater.com/auto: "true"`.
   Note: OTel collectors are intentionally **not** annotated (see
   [Gotchas](#gotchas)).

## Gotchas

- **`kustomization_resource`, not `kubernetes_manifest`.** `kubernetes_manifest`
  runs a server-side dry-run at **plan** time against the `external-secrets.io`
  CRDs; that fails when the CRD isn't installed yet (and these CRDs are created by
  the ESO Helm release in this very stack). `kustomization_resource` defers
  reconciliation to apply time and honors `depends_on`.
- **`remoteRef.key` is the item title; the field goes in `remoteRef.property`** on
  the `onepassword` (Connect) provider (see runbook step 2). The old `onepasswordSDK`
  provider slash-encoded the field into `key` with no `property` ŌĆö that grammar no
  longer applies.
- **`creationPolicy: Owner`** means ESO will overwrite/delete a Secret of the
  same name it didn't create ŌĆö don't point an ExternalSecret at a hand-managed
  Secret.
- **OTel collectors are not reloader-annotated.** They're managed by the OTel
  operator; annotating them would fight the operator. Operator-native rotation is
  a future task.

## Verify

```bash
# store should be Valid/Ready
kubectl get clustersecretstore onepassword

# the ExternalSecret should report SecretSynced
kubectl -n <namespace> get externalsecret <name>

# the produced Secret should exist with the expected keys
kubectl -n <namespace> get secret <target.name> -o jsonpath='{.data}' | tr ',' '\n'
```

## Notes

- **GitOps standard.** This stack normally applies via CI on push to `main`
  (`deploy-external-secrets` job). Local `tofu apply` is also safe ŌĆö the Vultr S3
  backend has `use_lockfile` enabled, so state locking serializes writes.
